# 커피숍 주문 시스템 요구사항

## Summary

- **기능 개요:** 커피 메뉴를 조회하고, 사용자가 포인트를 충전하여 커피를 주문·결제하며, 최근 주문 내역을 기준으로 인기 메뉴를 조회할 수 있는 시스템을 제공한다.
- **비즈니스 목표:** 다수 서버와 다수 인스턴스 환경에서도 주문, 포인트, 인기 메뉴 집계가 안정적이고 일관되게 동작하도록 한다.
- **기대 결과:** 메뉴 조회, 포인트 충전, 주문·결제, 인기 메뉴 조회 기능과 각 기능 및 제약사항을 검증하는 테스트가 제공된다.
- **요구사항 출처:** 사용자가 제공한 「커피숍 주문 시스템」 과제 원문.
- **문서 범위:** 이 문서는 과제 원문에서 확인된 요구사항과 미확정 사항만 다룬다. 기술 선택, API 상세 명세, ERD, 구현 코드는 범위에서 제외한다.

## Functional Requirements

### FR-1. 커피 메뉴 목록 조회

- 시스템은 커피 주문에 사용할 수 있는 메뉴 목록을 조회할 수 있어야 한다.
- 각 메뉴의 메뉴 ID, 이름, 가격을 제공해야 한다.

### FR-2. 포인트 충전

- 시스템은 인증된 사용자 정보에서 사용자 식별값을 추출하고, 충전금액을 입력받아 해당 사용자의 포인트를 충전해야 한다.
- 사용자 식별값을 요청 Body에서 직접 입력받지 않아야 한다.
- 충전금액은 `1원 = 1P`의 비율로 포인트에 반영되어야 한다.
- 충전금액은 1 이상의 정수여야 하며 업무상 최대 한도는 두지 않는다. 자료형 범위를 벗어나는 값과 존재하지 않거나 유효하지 않은 사용자의 요청은 거부해야 한다.

### FR-3. 커피 주문 및 결제

- 시스템은 인증된 사용자 정보와 하나 이상의 주문 항목을 기준으로 커피 주문과 결제를 진행해야 한다.
- 각 주문 항목은 메뉴 ID와 수량을 포함해야 한다.
- 사용자 식별값을 요청 Body에서 직접 입력받지 않아야 한다.
- 주문금액은 서버가 주문 시점의 현재 메뉴 가격과 각 주문 항목의 수량을 기준으로 계산해야 한다.
- 주문 시점의 메뉴 단가는 각 주문 항목에 스냅샷으로 보존되어야 한다.
- 결제는 사용자가 충전한 포인트로만 가능해야 한다.
- 결제가 이루어지면 사용자의 충전 포인트에서 주문금액을 차감해야 한다.
- 존재하지 않거나 주문할 수 없는 메뉴가 포함되거나 포인트 잔액이 부족하면 주문 생성과 포인트 차감이 모두 이루어지지 않아야 한다.
- 주문 생성, 포인트 차감과 이벤트 발행 의도 기록은 하나의 로컬 데이터베이스 트랜잭션에서 함께 성공하거나 실패해야 한다.
- 주문 취소와 결제 취소·환불은 현재 범위에서 지원하지 않는다.

### FR-4. 주문 내역 실시간 전송

- 시스템은 주문 내역을 데이터 수집 플랫폼으로 실시간 전송해야 한다.
- 전송 데이터에는 인증된 사용자의 식별값, 복수 주문 항목과 총 결제금액이 포함되어야 한다.
- 각 전송 주문 항목은 메뉴 ID와 수량을 표현할 수 있어야 한다.
- 과제 검증을 위한 데이터 수집 플랫폼은 Mock API 또는 테스트 코드로 대체할 수 있다.

### FR-5. 인기 메뉴 목록 조회

- 시스템은 조회 시점 기준 최근 168시간 동안의 인기 메뉴를 최대 3개까지 조회할 수 있어야 한다.
- 인기 메뉴 순위는 메뉴별 총 판매 수량 내림차순으로 결정해야 한다.
- 총 판매 수량이 같으면 해당 메뉴가 포함된 주문 건수 내림차순, 메뉴 ID 오름차순으로 순위를 결정해야 한다.
- 집계된 메뉴가 3개 미만이면 실제 집계된 메뉴만 반환해야 한다.
- 정상 상황에서는 주문 완료 후 10초 이내에 해당 판매 수량이 인기 메뉴 결과에 반영되어야 한다.
- 인기 메뉴 산정에 사용되는 메뉴별 총 판매 수량과 주문 건수는 정확해야 한다.
- 업무 기준 시간대는 `Asia/Seoul`로 하고 저장·이벤트 시각은 UTC로 관리하며, 달력 날짜가 아닌 조회 시점부터 정확히 168시간 전까지의 이동 구간을 사용해야 한다.
- 결제까지 성공한 `OrderCompleted` 주문만 집계하고 결제 실패 등 완료되지 않은 주문은 제외해야 한다.

## Non-Functional Requirements

### NFR-1. 확장성

- 애플리케이션은 다수 서버에 다수 인스턴스로 동작하는 환경에서도 요구 기능에 문제가 없도록 설계·구현되어야 한다.

### NFR-2. 동시성

- 포인트 충전, 주문·결제, 인기 메뉴 집계를 포함한 기능은 동시성 이슈를 고려하여 설계·구현되어야 한다.

### NFR-3. 데이터 일관성

- 포인트, 주문·결제 내역, 주문 항목별 가격 스냅샷, 메뉴별 판매 수량과 주문 건수는 데이터 일관성을 고려하여 처리되어야 한다.
- 다수 서버 및 동시 요청 상황에서도 메뉴별 판매 수량과 주문 건수의 정확성이 유지되어야 한다.

### NFR-4. 신뢰성

- 시스템은 다수 서버 환경에서도 안정적으로 동작해야 한다.

### NFR-5. 전송 적시성

- 주문 내역은 데이터 수집 플랫폼으로 실시간 전송되어야 한다.
- 정상 상황에서 주문 완료 후 Kafka 발행까지의 목표는 3초 이내다.
- 주문 트랜잭션에 기록된 Outbox 발행 의도는 일시적인 Producer 장애 시 지수 백오프로 최대 24시간 재시도하고, 영구적 장애는 실패 상태로 분류해야 한다.
- Kafka 발행 또는 데이터 수집 플랫폼 전달 실패는 이미 완료된 주문·결제의 성공 여부를 바꾸지 않아야 한다.
- 데이터 수집 플랫폼 전달 완료까지의 end-to-end 최대 지연시간은 측정 자료가 없어 아직 확정하지 않는다.
- Consumer Group별 Retry Topic과 DLT를 분리하여 일시 실패가 원래 Topic의 후속 주문 처리를 막지 않게 해야 한다.

### NFR-6. 인기 메뉴 반영 적시성

- 정상 상황에서는 주문 완료 후 10초 이내에 판매 수량이 인기 메뉴 결과에 반영되어야 한다.
- 반영 시간은 주문 이벤트 전달, PostgreSQL 집계 반영과 Redis 캐시 무효화 후 조회가 최신 결과를 반환할 수 있는 시점까지를 기준으로 측정해야 한다.
- 10초 이내 반영은 주문 이벤트를 소비한 PostgreSQL 집계 반영과 Redis 캐시 무효화로 충족하고, Redis 인기 메뉴 캐시의 정기 갱신 주기는 10초보다 길게 설정해야 한다.
- PostgreSQL 집계 반영 후 Redis 캐시 무효화에 실패하면 Offset을 확정하지 않고, 재전달 시 집계는 중복 반영하지 않은 채 캐시 무효화를 다시 시도해야 한다.

### NFR-7. 구현 재량 및 제약

- 인프라 사용에는 제약이 없다.
- API의 HTTP Method는 `GET`, `POST`, `PUT`, `DEL` 중 자유롭게 선택할 수 있다.
- 에러 응답과 에러 코드는 자유롭게 정의할 수 있다.

### NFR-8. 서비스 시간대 기반 성능 검증

- 성능 검증은 카페 주문 서비스의 평시, 아침 시간대 피크, 점심 시간대 피크와 순간 주문 집중 구간을 구분해야 한다.
- 각 구간은 요청량뿐 아니라 주문과 조회의 비율, 특정 사용자·메뉴에 요청이 집중되는 조건을 포함해야 한다.
- 응답시간 목표는 각 부하 구간별 `p95`, `p99`, 오류율과 함께 정의하고 측정 환경 및 테스트 데이터 조건을 기록해야 한다.
- 목표값 없이 수행한 기준 성능 측정 결과를 수집하되, 현재 구현이 낸 수치를 그대로 응답시간 목표로 채택해서는 안 된다.
- 최종 응답시간 목표는 사용자 관점의 허용 지연, 서비스 시간대별 예상 부하, 기준 성능과 피크 대응 여유 용량을 함께 고려하여 결정해야 한다.
- 구간별 요청량과 응답시간 수치는 근거가 확인되기 전까지 확정값으로 간주하지 않는다.

## Project Requirements

### PR-1. 검증 테스트

- 각 기능과 제약사항에 대한 테스트를 작성해야 한다.

### PR-2. 설계 및 문제 해결 전략 문서화

- `README.md`에 설계 내용(ERD, API 명세서), 설계 의도, 선택한 문제 해결 전략과 그 선택을 위해 분석한 내용, 기술적 선택 이유를 포함해야 한다.
- Kafka를 선택한 이유는 주문 처리와 후속 처리의 분리, 피크 트래픽 완충, 독립적인 Consumer 확장과 이벤트 재처리 요구에 연결하여 설명해야 한다.
- Kafka에 발행하는 이벤트별 Producer, Consumer와 소비 후 수행하는 업무를 연결하여 설명해야 한다.
- 주문 완료 이벤트를 소비하는 데이터 수집 플랫폼 전달 Consumer와 인기 메뉴 집계 Consumer의 책임을 구분해야 한다.
- Kafka Consumer의 업무별 Consumer Group, 파티션과 Consumer 인스턴스의 관계, Offset 확정, 재시도·DLT 및 멱등 처리 경계를 설명해야 한다.
- Redis 고가용성 학습 범위에는 Primary/Replica와 Sentinel 기반 장애 감지·자동 Failover 구조 및 장애 중 PostgreSQL 대체 조회 경로를 포함해야 한다. 실제 배포 범위는 별도 의사결정으로 남긴다.
- PostgreSQL 고유 기능을 실제 동시성 또는 데이터 처리 문제 해결에 적용하고, 일반적인 조회·저장 용도를 넘어 해당 기능을 선택한 이유와 검증 결과를 설명해야 한다.
- 위 산출물의 실제 내용 작성은 이 요구사항 문서의 범위에 포함하지 않는다.

## Business Rules

### BR-1. 메뉴 정보

- 조회 가능한 각 커피 메뉴는 메뉴 ID, 이름, 가격 정보를 가진다.

### BR-2. 포인트 환산

- 포인트 충전 시 `1원 = 1P`를 적용한다.
- 충전금액은 1 이상의 정수이며 업무상 최대 한도는 두지 않는다.
- 포인트는 만료되지 않으며 사용자 간 양도를 지원하지 않는다.

### BR-3. 결제 수단

- 커피 주문의 결제 수단은 포인트로 한정한다.
- 주문 결제금액은 사용자가 충전한 포인트에서 차감한다.
- 주문 결제금액은 서버가 주문 시점의 현재 메뉴 가격과 주문 수량으로 계산한다.
- 주문 시점의 메뉴 단가는 주문 항목에 스냅샷으로 보존한다.
- 잔액이 주문금액보다 부족하면 주문을 생성하거나 포인트를 차감하지 않는다.
- 주문 취소와 결제 취소·환불은 현재 범위에서 지원하지 않는다.

### BR-4. 인기 메뉴 범위

- 인기 메뉴 조회 대상 기간은 조회 시점 기준 최근 168시간이다.
- 인기 메뉴 조회 결과는 최대 3개 메뉴로 구성하며, 집계된 메뉴가 3개 미만이면 실제 집계된 메뉴만 반환한다.
- 인기 메뉴는 메뉴별 총 판매 수량 내림차순으로 결정한다.
- 총 판매 수량이 같으면 해당 메뉴가 포함된 주문 건수 내림차순, 메뉴 ID 오름차순으로 결정한다.
- 인기 메뉴 산정의 기반이 되는 메뉴별 총 판매 수량과 주문 건수는 정확해야 한다.
- 정상 상황에서는 주문 완료 후 10초 이내에 판매 수량이 인기 메뉴 결과에 반영되어야 한다.
- 업무 기준 시간대는 `Asia/Seoul`이며 저장·이벤트 시각은 UTC로 관리한다.
- 조회 시점부터 정확히 168시간 전까지의 이동 구간에서 결제까지 성공한 `OrderCompleted` 주문만 집계한다.

### BR-5. 주문 내역 전송 데이터

- 데이터 수집 플랫폼에 전송하는 주문 내역에는 인증된 사용자의 식별값, 복수 주문 항목과 총 결제금액이 포함되어야 한다.
- 각 전송 주문 항목은 메뉴 ID와 수량을 표현할 수 있어야 한다.

### BR-6. 주문 구성과 사용자 식별

- 한 번의 주문은 하나 이상의 메뉴를 포함할 수 있으며, 각 메뉴의 주문 수량을 지정할 수 있다.
- 포인트 충전과 주문에 사용하는 사용자 식별값은 인증된 사용자 정보에서 추출하며 요청 Body에서 직접 받지 않는다.

## Dependencies

### DEP-1. 메뉴 정보

- 메뉴 목록 조회, 주문금액 계산과 주문 항목의 가격 스냅샷 보존을 위해 메뉴 ID, 이름, 현재 가격을 포함한 메뉴 정보가 필요하다.

### DEP-2. 사용자 식별값 및 포인트 정보

- 포인트 충전과 주문·결제를 연결하기 위해 인증된 사용자 정보에서 추출한 식별값과 해당 사용자의 충전 포인트 정보가 필요하다.

### DEP-3. 주문 내역

- 인기 메뉴 산정과 데이터 수집 플랫폼 전송을 위해 주문 시각, 사용자 식별값, 총 결제금액과 메뉴 ID·수량·주문 시점 단가를 포함한 주문 항목 내역이 필요하다.

### DEP-4. 데이터 수집 플랫폼

- 주문 내역을 실시간으로 전달받을 데이터 수집 대상이 필요하다.
- 과제 범위에서는 Mock API 또는 테스트 코드로 대체할 수 있다.
- 실제 플랫폼의 인터페이스, 가용성, 장애 처리 계약은 제공되지 않았다.

### DEP-5. 인증된 사용자 정보

- 포인트 충전과 주문 처리에서 사용자 식별값을 요청 Body가 아닌 인증된 사용자 정보로부터 제공받을 수 있어야 한다.
- 인증 정보의 제공 주체와 구체적인 검증 계약은 제공되지 않았다.

## Acceptance Criteria

### AC-1. 메뉴 조회

- 메뉴 목록을 조회하면 각 메뉴의 메뉴 ID, 이름, 가격을 확인할 수 있다.

### AC-2. 포인트 충전

- 인증된 사용자 정보와 충전금액으로 충전하면 해당 사용자의 포인트가 `1원 = 1P` 비율로 증가한다.
- 포인트 충전 요청 Body에서 사용자 식별값을 직접 받지 않는다.
- 0 이하, 정수가 아닌 값, 자료형 범위를 벗어난 값과 존재하지 않거나 유효하지 않은 사용자의 충전 요청은 거부된다.

### AC-3. 주문 및 포인트 결제

- 인증된 사용자 정보와 하나 이상의 메뉴 ID·수량으로 주문·결제가 완료되면 주문 및 주문 항목 내역이 생성되고, 해당 사용자의 충전 포인트에서 주문금액이 차감된다.
- 주문금액은 서버가 주문 시점의 현재 메뉴 가격과 수량으로 계산한 금액과 일치한다.
- 각 주문 항목에는 주문 시점의 메뉴 단가가 스냅샷으로 보존된다.
- 주문 요청 Body에서 사용자 식별값을 직접 받지 않는다.
- 커피 주문은 포인트 이외의 결제 수단으로 결제되지 않는다.
- 주문할 수 없는 메뉴가 포함되거나 잔액이 부족하면 주문, 주문 항목, 포인트 차감과 Outbox 기록이 부분적으로 남지 않는다.
- 동일한 일회성 주문 토큰의 재시도는 주문과 포인트 차감을 중복 발생시키지 않고 확정된 동일 결과를 반환한다.

### AC-4. 주문 내역 전송

- 주문 내역의 인증된 사용자 식별값, 복수 주문 항목과 총 결제금액이 데이터 수집 플랫폼 또는 이를 대체하는 Mock API·테스트 코드로 실시간 전송된 사실을 확인할 수 있다.
- 전송된 각 주문 항목에서 메뉴 ID와 수량을 확인할 수 있다.
- Consumer 처리 실패가 Retry Topic을 거쳐 재시도되고 한도를 넘으면 해당 Consumer Group의 DLT로 격리되며, 재처리 시 기존 `eventId`가 유지되어 업무 효과가 중복되지 않는다.

### AC-5. 인기 메뉴 조회

- 조회 시점 기준 최근 168시간의 집계 대상 메뉴가 3개 이상이면 인기 메뉴 3개가 제공된다.
- 집계 대상 메뉴가 3개 미만이면 실제 집계된 메뉴만 제공되고, 집계 대상이 없으면 빈 목록이 제공된다.
- 인기 메뉴는 총 판매 수량 내림차순으로 정렬되고, 동률이면 해당 메뉴가 포함된 주문 건수 내림차순, 메뉴 ID 오름차순으로 정렬된다.
- 조회 결과 산정에 사용된 메뉴별 총 판매 수량과 주문 건수가 실제 집계 대상 주문 항목 내역과 일치한다.
- 정상 상황에서는 주문 완료 후 10초 이내에 해당 판매 수량이 인기 메뉴 결과에 반영된다.
- `Asia/Seoul` 업무 시간대와 UTC 저장 시각을 기준으로 조회 시점부터 정확히 168시간 전까지 결제 완료 주문만 집계된다.
- PostgreSQL 집계 성공 후 Redis 무효화가 실패하면 이벤트 재전달에서 집계는 중복 반영되지 않고 캐시 무효화가 다시 시도된다.

### AC-6. 다중 인스턴스와 동시성

- 다수 서버의 다수 인스턴스에서 동시 요청이 발생해도 포인트, 주문·결제 내역, 주문 항목별 가격 스냅샷, 메뉴별 판매 수량과 주문 건수에 유실이나 불일치가 발생하지 않는다.

### AC-7. 테스트

- 메뉴 조회, 인증된 사용자 기반 포인트 충전, 복수 주문 항목의 주문·결제, 서버 가격 계산, 주문 단가 스냅샷, 복수 주문 항목 전송, 인기 메뉴의 판매 수량·정렬·반환·반영 기준과 동시성·데이터 일관성 제약을 검증하는 테스트가 존재한다.
- 평시, 아침 시간대 피크, 점심 시간대 피크와 순간 주문 집중 구간별로 처리량, `p95`·`p99` 응답시간, 오류율, Consumer Lag과 인기 메뉴 반영 시간을 확인할 수 있다.
- 기준 성능 측정 결과와 이를 근거로 확정한 구간별 성능 Threshold를 구분하여 확인할 수 있다.
- 다중 Outbox Publisher가 `FOR UPDATE SKIP LOCKED`로 동일 이벤트를 중복 선점하지 않는지, 중복 Kafka 이벤트가 Unique Constraint와 `ON CONFLICT`에 의해 업무 효과를 중복 발생시키지 않는지 검증하는 테스트가 존재한다.
- Outbox Partial Index 적용 전후 실행 계획 또는 처리 비용 비교 결과를 확인할 수 있다.

### AC-8. 설계 문서

- `README.md`에서 ERD, API 명세서, 설계 의도, 문제 해결 전략과 분석 내용, 기술적 선택 이유를 확인할 수 있다.
- Kafka 사용 이유, 이벤트별 Producer·Consumer와 소비 책임, Redis 고가용성 학습 결과, PostgreSQL 고유 기능의 적용 이유와 검증 결과를 확인할 수 있다.
