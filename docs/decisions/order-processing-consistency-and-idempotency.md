# 주문 처리, 포인트 일관성 및 멱등성

## Status

Accepted

## Context

- 관련 요구사항: `FR-2 — 포인트 충전`, `FR-3 — 커피 주문 및 결제`, `NFR-2 — 동시성`, `NFR-3 — 데이터 일관성`, `BR-2 — 포인트 환산`, `BR-3 — 결제 수단`, `BR-6 — 주문 구성과 사용자 식별`, `DEP-1 — 메뉴 정보`, `DEP-2 — 사용자 식별값 및 포인트 정보`, `DEP-5 — 인증된 사용자 정보`, `AC-2 — 포인트 충전`, `AC-3 — 주문 및 포인트 결제`, `AC-6 — 다중 인스턴스와 동시성`.
- 주문은 복수 항목의 가격을 서버가 확정하고 포인트를 차감해야 하며, 다중 인스턴스와 재시도 상황에서도 중복 주문이나 중복 차감이 없어야 한다.
- 이 문서는 API 필드나 데이터 모델 상세가 아니라 주문 한 건의 업무적·트랜잭션적 경계를 기록한다.

## Decision

- 주문은 하나 이상의 주문 항목으로 구성하고 각 항목은 메뉴 ID와 수량을 가진다.
- 주문금액은 서버가 주문 시점의 현재 메뉴 가격과 수량으로 계산하며, 사용한 메뉴 단가는 주문 항목에 스냅샷으로 보존한다.
- 사용자 식별값은 요청 Body에서 받지 않고 인증된 사용자 정보에서 추출한다. 과제 범위에서는 외부 인증 정보의 검증까지만 다루고 전체 인증 시스템은 구현하지 않는다.
- 결제 수단은 포인트로 한정한다. 충전금액은 1 이상의 정수로 받고 업무상 최대 한도는 두지 않으며, 자료형 범위를 벗어나는 값은 기술적으로 거부한다.
- 존재하지 않거나 유효하지 않은 사용자의 요청은 거부한다. 잔액이 부족하면 주문을 생성하거나 포인트를 차감하지 않으며, 충전 후에는 새 주문으로 다시 시도한다.
- 포인트 충전은 현재 잔액을 애플리케이션에서 읽어 다시 저장하지 않고 DB의 원자적 증가 연산으로 처리한다.
- 포인트 차감은 잔액이 주문금액 이상일 때만 성공하는 조건부 원자적 감소 연산으로 처리한다. 갱신 성공 여부로 결제 성공과 잔액 부족을 구분한다.
- 충전과 차감이 동시에 실행되면 DB에 먼저 반영된 연산을 기준으로 후속 연산의 성공 여부를 결정한다. 충전 포인트를 주문에 확실히 사용하려는 클라이언트는 충전 완료 응답 후 주문해야 한다.
- 동일 주문 토큰의 동시 요청은 토큰의 조건부 상태 변경으로 처리 권한을 하나의 요청에만 부여한다.
- 처리 권한을 얻은 요청은 포인트 조건부 차감, 주문·주문 항목 생성, 이벤트 발행 의도 기록과 토큰 결과 확정을 하나의 로컬 DB 트랜잭션에서 함께 성공시키거나 실패시킨다. 외부 이벤트의 실제 발행 성공은 주문 성공 조건에 포함하지 않는다.
- 서버가 일회성 주문 토큰을 발급한다. 토큰 하나는 주문 시도 하나에 대응하고, 확정된 성공 또는 업무 실패 결과를 재시도에도 동일하게 반환한다.
- 사용하지 않은 주문 토큰의 유효기간은 10분, 확정된 처리 결과의 보관기간은 24시간으로 한다.
- 메뉴는 현재 주문 가능 여부를 검증하되 원재료별 수량과 레시피 재고는 현재 범위에서 모델링하지 않는다.
- 주문 취소와 결제 취소·환불은 현재 범위에서 지원하지 않는다.
- 포인트는 만료되지 않으며 사용자 간 양도를 지원하지 않는다.

이 방향은 가격과 사용자 식별을 서버가 통제하고 주문과 포인트 변경의 원자성을 유지하여 위·변조, 초과 차감, 부분 성공을 줄인다. 대신 서버가 토큰 상태와 처리 결과를 보관해야 하고, 동시성 제어와 트랜잭션 충돌을 검증해야 한다.

## Options Considered

- Option: 서버 권위의 복수 항목 주문과 포인트 원자 결제.
- Rationale: `FR-3 — 커피 주문 및 결제`, `NFR-2 — 동시성`, `NFR-3 — 데이터 일관성`의 가격 정확성, 포인트 일관성과 다중 인스턴스 요구를 함께 만족한다.
- Trade-offs: 로컬 트랜잭션의 책임과 동시성 검증 범위가 커진다.
- Decision: Accepted.

- Option: 서버가 발급하는 일회성 주문 토큰으로 재시도를 식별한다.
- Rationale: 클라이언트가 임의 키의 생성·수명·재사용 규칙을 관리하지 않아도 주문 시도를 명확하게 식별할 수 있다.
- Trade-offs: 토큰 발급 단계와 토큰·처리 결과 저장 공간이 추가된다.
- Decision: Accepted.

### 포인트 충전 동시성 전략

| 전략 | 정합성 | 성능 | 구현 복잡도 | 다중 서버 적합성 | 결정 |
| --- | --- | --- | --- | --- | --- |
| DB 원자적 증가 | DB 행 갱신으로 Lost Update 방지 | 별도 조회·외부 락이 없어 좋음 | 낮음 | 높음 | Accepted |
| 비관적 락 | 높음 | 같은 사용자 요청이 락을 기다림 | 보통 | 높음 | 검토됨, Rejected 아님 |
| 낙관적 락 | 충돌 재시도 시 높음 | 경합이 낮을 때 좋고 높을 때 재시도 증가 | 보통~높음 | 높음 | 검토됨, Rejected 아님 |
| Redis 분산 락 | 락이 유지될 때 직렬화 가능 | Redis 왕복과 락 경합 추가 | 높음 | 높음 | 검토됨, Rejected 아님 |
| 애플리케이션 로컬 락 | 단일 인스턴스 안에서만 보장 | 인스턴스 내부에서는 좋음 | 낮음 | 낮음 | 검토됨, Rejected 아님 |

- Rationale: 충전은 현재 잔액에 양수를 더하는 단순 연산이므로 PostgreSQL의 원자적 행 갱신만으로 다중 서버의 Lost Update를 방지할 수 있다.
- Trade-offs: 충전과 차감의 동시 실행 순서를 별도로 보장하지 않으므로, 충전 결과를 주문에 사용하려면 충전 완료 후 주문해야 한다.
- Decision: DB 원자적 증가를 Accepted로 선택했다.

### 포인트 차감 동시성 전략

| 전략 | 정합성 | 성능 | 구현 복잡도 | 다중 서버 적합성 | 결정 |
| --- | --- | --- | --- | --- | --- |
| 잔액 조건부 원자적 감소 | 잔액 확인과 차감을 한 연산으로 수행해 이중 사용 방지 | 별도 락 관리가 없어 좋음 | 낮음 | 높음 | Accepted |
| 비관적 락 | 높음 | 동일 사용자 주문이 순차 대기 | 보통 | 높음 | 검토됨, Rejected 아님 |
| 낙관적 락 | 전체 트랜잭션 재시도 시 높음 | 충돌이 많으면 재시도 비용 증가 | 높음 | 높음 | 검토됨, Rejected 아님 |
| Redis 분산 락 | Redis 락과 DB 트랜잭션이 분리됨 | 외부 왕복과 장애 지점 추가 | 높음 | 높음 | 검토됨, Rejected 아님 |
| 애플리케이션 로컬 락 | 다른 인스턴스의 차감을 막지 못함 | 인스턴스 내부에서는 좋음 | 낮음 | 낮음 | 검토됨, Rejected 아님 |

- Rationale: 잔액 확인과 차감을 하나의 DB 연산으로 결합하면 음수 잔액과 동시 이중 사용을 방지하면서 주문 트랜잭션에 포함할 수 있다.
- Trade-offs: 같은 사용자에게 차감 요청이 집중되면 해당 DB 행에서 경합하지만, 명시적 장기 락이나 별도 Redis 락은 필요하지 않다.
- Decision: 잔액 조건부 원자적 감소를 Accepted로 선택했다.

### 주문 트랜잭션 동시성 전략

| 전략 | 정합성 | 성능 | 구현 복잡도 | 다중 서버 적합성 | 결정 |
| --- | --- | --- | --- | --- | --- |
| 주문 토큰 조건부 상태 변경 | 한 요청만 처리 권한을 얻어 중복 주문·차감 방지 | 짧은 DB 경합으로 처리 가능 | 보통 | 높음 | Accepted |
| 주문 토큰 비관적 락 | 높음 | 같은 토큰 요청이 트랜잭션 종료까지 대기 | 보통 | 높음 | 검토됨, Rejected 아님 |
| 주문 토큰 낙관적 락 | 재시도와 결과 재조회 시 높음 | 충돌 시 전체 주문 재시도 비용 발생 | 높음 | 높음 | 검토됨, Rejected 아님 |
| Redis 분산 락 | DB 제약을 함께 사용해야 안전 | Redis 왕복과 임대 만료 처리 추가 | 높음 | 높음 | 검토됨, Rejected 아님 |
| 애플리케이션 로컬 락 | 다른 인스턴스의 동일 토큰 요청을 막지 못함 | 인스턴스 내부에서는 좋음 | 낮음 | 낮음 | 검토됨, Rejected 아님 |

- Rationale: 이미 확정된 일회성 주문 토큰을 처리 권한의 기준으로 사용하면 동일 토큰의 동시 요청을 DB에서 직접 제어하고 주문의 로컬 변경을 하나의 트랜잭션으로 묶을 수 있다.
- Trade-offs: 업무 실패도 토큰의 확정 결과로 남겨야 하며, 조건부 상태 변경에 실패한 재시도 요청이 기존 처리 결과를 조회하는 흐름이 필요하다.
- Decision: 주문 토큰 조건부 상태 변경과 단일 로컬 DB 트랜잭션을 Accepted로 선택했다.

## Proposed

- 동일 토큰에 다른 요청 내용이 들어오면 충돌 오류로 처리하고, 오류 응답의 구체적인 형식은 Specification에서 정의한다.
- 공통 응답 형식을 사용하되 필드 구성과 오류 코드 체계는 Specification 단계에서 확정한다.
- 메뉴 주문 가능 상태의 구체적인 상태값과 전이 규칙은 도메인 및 Specification 단계에서 정의한다.

## Rejected Alternatives

- Alternative: 요청 Body에서 사용자 식별값을 직접 받는다.
- Reason: 인증된 사용자 정보에서 식별값을 추출하기로 명시적으로 확정했으며 `FR-2 — 포인트 충전`, `FR-3 — 커피 주문 및 결제`, `BR-6 — 주문 구성과 사용자 식별`과 충돌한다.

- Alternative: 원재료별 초기 재고와 레시피 수량까지 현재 과제 범위에서 관리한다.
- Reason: 카페의 실제 재고 조건이 별도 요구사항 없이 고정되기 어렵고, 현재 범위에서는 메뉴 주문 가능 여부만 관리하기로 명시적으로 정했다.

- Alternative: 주문 취소와 결제 취소·환불을 현재 범위에 포함한다.
- Reason: 과제 요구사항에 없으므로 현재 범위에서는 지원하지 않기로 명시적으로 정했다.

- Alternative: 포인트 만료 또는 사용자 간 양도를 현재 범위에 포함한다.
- Reason: 현재 과제의 주문·결제 핵심 범위를 넘어 별도의 만료 배치, 소멸 이력과 사용자 간 이전 규칙이 필요하므로 지원하지 않기로 명시적으로 정했다.

## Downstream Impact

- Domain: Order, OrderItem, PointBalance, OrderToken의 책임과 주문 성공 경계를 정의해야 한다.
- Specification: 복수 주문 항목, 인증 사용자 식별, 가격 스냅샷, 토큰 발급·사용·재시도, 잔액 부족과 유효성 오류 및 포인트 비만료·양도 불가를 명세해야 한다.
- Issue: 주문 모델, 포인트 동시성, 토큰 수명주기와 실패 시나리오를 추적 가능한 구현 작업으로 분해해야 한다.
- Implementation: 포인트 충전은 원자적 증가, 차감은 잔액 조건부 원자적 감소로 처리하고, 주문 토큰의 조건부 상태 변경으로 처리 권한을 획득해야 한다. 주문·포인트·이벤트 발행 의도와 토큰 결과는 단일 로컬 트랜잭션으로 처리하되 구체적인 클래스 구조와 SQL은 이 문서에서 정하지 않는다.
- Test: 복수 항목 합계, 단가 스냅샷, 동시 충전·차감, 잔액 부족, 동일 토큰 동시 요청, 다중 인스턴스 중복 효과와 지원하지 않는 포인트 양도 경계를 검증해야 한다.
- Pull Request / Review: 변경이 주문 성공 경계, 포인트 원자성 또는 멱등성 규칙을 약화하지 않는지 확인해야 한다.

## Assumptions

- 추가로 채택한 미검증 가정은 없다.

## Open Decisions

- 동일한 메뉴 ID가 여러 주문 항목으로 들어오면 합산할지 거부할지.
- 메뉴별 수량과 주문당 항목 수의 상한.
- 인증 정보 제공 주체, 토큰 형식과 필수 Claim.
- 주문 토큰과 처리 결과의 정리 주기 및 장기 감사 데이터 보관 정책.
- 만료되거나 존재하지 않는 토큰과 동일 토큰·다른 요청을 구분하는 구체적인 오류 계약.
