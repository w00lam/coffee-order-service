# 인기 메뉴 집계와 조회

## Status

Accepted

## Context

- 관련 요구사항: `FR-5 — 인기 메뉴 목록 조회`, `NFR-2 — 동시성`, `NFR-3 — 데이터 일관성`, `NFR-6 — 인기 메뉴 반영 적시성`, `BR-4 — 인기 메뉴 범위`, `DEP-3 — 주문 내역`, `AC-5 — 인기 메뉴 조회`, `AC-6 — 다중 인스턴스와 동시성`.
- 인기 메뉴는 최근 168시간의 정확한 판매 수량과 주문 건수를 사용하면서 정상 상황에서 주문 완료 후 10초 이내에 갱신되어야 한다.
- 다중 인스턴스에서 집계 정확성과 조회 성능을 함께 유지할 읽기 모델이 필요하다.
- 2026-07-14 피드백에 따라 Redis 캐시를 5초마다 갱신한다는 하위 결정은 Superseded로 전환한다. 주문 후 10초 이내 반영은 Kafka Consumer의 PostgreSQL 집계 반영과 Redis 캐시 무효화가 담당하고, Redis 인기 메뉴 캐시의 정기 Refresh 주기는 10초보다 길게 둔다.

## Decision

- 조회 시점 기준 최근 168시간의 이동 구간을 집계한다.
- 메뉴별 총 판매 수량 내림차순으로 정렬하고, 동률이면 해당 메뉴가 포함된 주문 건수 내림차순, 메뉴 ID 오름차순으로 정렬한다.
- 최대 3개를 반환한다. 집계된 메뉴가 3개 미만이면 실제 집계 결과만 반환하고, 집계 결과가 없으면 빈 목록을 반환한다.
- 주문 이벤트를 소비하여 주문 항목별 수량과 메뉴가 포함된 주문 건수를 집계한다.
- 관계형 데이터베이스의 집계 데이터를 기준 데이터로 사용하고 Redis는 인기 메뉴 조회 캐시로 사용한다.
- 인기 메뉴 Consumer는 주문 이벤트를 PostgreSQL 영속 집계에 반영한 뒤 Redis 인기 메뉴 캐시를 무효화하여 정상 상황의 10초 이내 반영을 충족한다.
- 캐시가 무효화된 뒤 처음 발생한 인기 메뉴 조회는 PostgreSQL의 최신 집계 결과를 조회하여 응답하고 Redis 캐시를 다시 채운다.
- Redis 인기 메뉴 캐시의 정기 Refresh 주기는 10초보다 길게 설정하여 10초 이내 반영이 정기 폴링이 아니라 Kafka 기반 실시간 집계로 달성되도록 한다.
- 정기 Refresh는 여러 인스턴스 중 하나만 수행하도록 Redisson `RLock`을 사용한다. 락 획득에 실패한 인스턴스는 기다리지 않고 해당 주기의 Refresh를 건너뛴다.
- 정확한 정기 Refresh 주기는 10초보다 긴 후보를 대상으로 PostgreSQL 조회 부하, 캐시 적중률과 운영 비용을 비교하여 결정한다.
- Redis 장애 시 관계형 데이터베이스의 집계 결과로 조회를 대체한다.
- 정상 상황에서는 주문 완료 후 10초 이내에 해당 판매 수량이 인기 메뉴 조회 결과에 반영되어야 한다.

이 방향은 정확성의 기준을 PostgreSQL 영속 집계 데이터에 두면서 Redis로 반복 조회 부하를 줄인다. Kafka Consumer의 집계 반영과 캐시 무효화가 10초 반영 목표를 담당하고, 10초보다 긴 정기 Refresh는 캐시 재적재를 담당한다. 대신 PostgreSQL 집계 반영과 Redis 무효화를 하나의 트랜잭션으로 묶을 수 없으므로 무효화 실패와 대체 조회 경로를 분리해 검증해야 한다.

## Options Considered

- Option: 관계형 데이터베이스 집계를 기준 데이터로 두고 Redis를 조회 캐시로 사용한다.
- Rationale: 집계 결과의 복구 가능성과 정확성을 유지하면서 인기 조회의 반복 부하를 줄일 수 있다.
- Trade-offs: 집계 저장 구조와 캐시 갱신·대체 조회 경로가 추가된다.
- Decision: Accepted.

- Option: 이벤트 기반 PostgreSQL 집계 및 Redis 캐시 무효화와 10초보다 긴 정기 Refresh를 함께 사용하고, 정기 Refresh는 Redisson `RLock`으로 조정한다.
- Rationale: Consumer가 주문 이벤트를 PostgreSQL에 10초 이내 반영하고 기존 Redis 캐시를 무효화하므로 다음 조회는 최신 집계 결과를 사용한다. 정기 Refresh를 10초보다 길게 두어 실시간성의 근거를 Kafka Consumer 처리로 분리하고, 다중 서버 중 한 인스턴스만 Refresh하여 중복 DB 조회 부하를 줄인다. Redisson이 락 소유권 확인과 안전한 해제를 제공하므로 직접 구현보다 복잡도가 낮다.
- Trade-offs: Redis 의존성이 추가되고 락 임대시간과 Watchdog 동작을 이해해야 한다. 락을 얻지 못한 인스턴스는 작업을 건너뛰므로 처리 대기는 없지만, 락 보유 인스턴스의 실패가 반복되면 캐시 갱신이 지연될 수 있다.
- Decision: Accepted.

| 캐시 Refresh 조정 전략 | 정합성 | 성능 | 구현 복잡도 | 다중 서버 적합성 | 결정 |
| --- | --- | --- | --- | --- | --- |
| Redisson `RLock` | 한 인스턴스만 Refresh하도록 조정 | 락 실패 시 즉시 건너뛰어 대기 없음 | 낮음~보통 | 높음 | Accepted |
| Redis `SET NX PX` 직접 구현 | 소유권 토큰과 원자적 해제가 정확하면 충분 | Redis 단일 명령 중심이라 좋음 | 보통 | 높음 | 검토됨, Rejected 아님 |
| PostgreSQL Advisory Lock | DB를 공통 조정 지점으로 사용 가능 | Refresh 동안 DB 연결을 점유 | 보통 | 높음 | 검토됨, Rejected 아님 |
| 애플리케이션 로컬 락 | 인스턴스별 중복만 방지 | 인스턴스 내부에서는 좋음 | 낮음 | 낮음 | 검토됨, Rejected 아님 |

## Proposed

- 이벤트 처리 중 캐시 무효화에 실패하면 Redis 조회를 우회하여 PostgreSQL 집계 결과를 제공한다.
- 조회 응답에 집계 기준 시각을 제공한다.
- 이벤트 집계와 원본 주문 내역의 차이를 탐지하기 위한 주기적 재집계를 둔다.

## Rejected Alternatives

- Alternative: 집계된 메뉴가 3개 미만일 때 판매되지 않은 메뉴를 추가하여 항상 3개를 반환한다.
- Reason: 실제 집계 결과만 반환하기로 명시적으로 확정했으며 `FR-5 — 인기 메뉴 목록 조회`, `BR-4 — 인기 메뉴 범위`, `AC-5 — 인기 메뉴 조회`와 충돌한다.

- Alternative: 집계된 메뉴가 1개 또는 2개일 때 오류나 무조건 빈 목록을 반환한다.
- Reason: 집계된 메뉴가 3개 미만이면 실제 결과를 그대로 반환하기로 명시적으로 확정했다.

## Downstream Impact

- Domain: 메뉴별 판매 수량, 주문 건수와 168시간 집계 기준의 의미를 정의해야 한다.
- Specification: 인기 메뉴 응답, 정렬, 최대 반환 수, 빈 결과와 반영 지연 기준을 명세해야 한다.
- Issue: 이벤트 소비 집계, 영속 집계 조회, Redis 갱신·락·대체 조회와 정합성 검증 작업을 분해해야 한다.
- Implementation: 동일 주문의 한 메뉴는 주문 건수에 한 번만 기여하면서 수량은 전체 판매 수량에 반영해야 한다. Consumer는 PostgreSQL 영속 집계 반영 후 Redis 인기 메뉴 캐시를 무효화해야 한다. Cache Miss 조회는 PostgreSQL 최신 집계 결과를 응답하고 Redis를 다시 채운다. 정기 Refresh는 Redisson `RLock`을 획득한 인스턴스만 수행하고 실패한 인스턴스는 해당 주기를 건너뛰되, 구체적인 클래스와 테이블·버킷 구조는 이 문서에서 정하지 않는다.
- Test: 168시간 경계, 수량·주문 건수 동률, 0~3개 반환, 중복 이벤트, 다중 Consumer, 이벤트 집계와 캐시 무효화를 통한 10초 이내 조회 반영, 다중 Refresh 인스턴스와 Redis 장애 시 정확성을 검증해야 한다. 10초보다 긴 후보 Refresh 주기별 PostgreSQL 조회 부하와 캐시 적중률을 비교해야 한다.
- Pull Request / Review: 주문 건수와 판매 수량의 혼용, 시간 경계 오류, 중복 집계와 캐시를 기준 데이터로 오인하는 변경을 확인해야 한다.

## Assumptions

- 추가로 채택한 미검증 가정은 없다.

## Open Decisions

- 최근 168시간을 계산할 기준 시간대.
- 결제 완료, 실패, 취소 또는 환불 상태 중 집계에 포함할 주문 상태.
- 집계 저장을 위한 시간 버킷 단위.
- 10초보다 긴 Redis 전체 Refresh 주기와 캐시 TTL.
- PostgreSQL 집계 반영 후 Redis 캐시 무효화에 실패했을 때 재시도 또는 PostgreSQL 대체 조회로 전환하는 구체적인 방식.
- Redisson `RLock` 임대시간과 Watchdog 사용 여부.
- 이벤트 집계와 원본 주문 데이터의 재검증 주기 및 보정 방식.
- 판매 중지되거나 삭제된 메뉴의 인기 결과 노출 여부.
