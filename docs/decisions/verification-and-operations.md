# 검증 및 운영 기준

## Status

Accepted

## Context

- 관련 요구사항: `NFR-1 — 확장성`, `NFR-2 — 동시성`, `NFR-3 — 데이터 일관성`, `NFR-4 — 신뢰성`, `NFR-5 — 전송 적시성`, `NFR-6 — 인기 메뉴 반영 적시성`, `PR-1 — 검증 테스트`, `PR-2 — 설계 및 문제 해결 전략 문서화`, `AC-1 — 메뉴 조회`, `AC-2 — 포인트 충전`, `AC-3 — 주문 및 포인트 결제`, `AC-4 — 주문 내역 전송`, `AC-5 — 인기 메뉴 조회`, `AC-6 — 다중 인스턴스와 동시성`, `AC-7 — 테스트`, `AC-8 — 설계 문서`.
- 다중 인스턴스, 동시성, 비동기 이벤트와 캐시 대체 경로는 정상 기능 테스트만으로 충분히 입증할 수 없다.
- 이 문서는 기능 구현 방식이 아니라 성공을 판정할 부하·신뢰성 기준과 검증 증거의 위치를 기록한다.
- 카페 주문 서비스는 평시와 아침·점심 시간대의 주문 집중도가 다르므로 하나의 고정 TPS와 응답시간만으로 실제 사용 패턴을 설명할 수 없다.
- 2026-07-14 피드백에 따라 근거 없이 확정했던 정상 100 TPS, 피크 500 TPS, 버스트 1,000 TPS와 응답시간 하위 결정은 Superseded로 전환하고, 서비스 시간대별 부하 모델과 기준 측정 결과로 정량 목표를 결정한다.

## Decision

- k6 부하 테스트를 평시, 아침 시간대 피크, 점심 시간대 피크와 순간 주문 집중 구간으로 나누어 구성한다.
- 각 구간에는 지속 시간, 단계별 요청량, 주문·조회 비율, 사용자·메뉴 분포와 Hot key 조건을 기록한다.
- 각 구간에서 처리량, `p95`·`p99` 응답시간, 오류율, PostgreSQL·Redis 부하, Outbox 적체와 Consumer Lag을 함께 측정한다.
- 먼저 Threshold 없이 동일 환경에서 반복 가능한 기준 성능을 측정하여 병목과 지연 분포를 파악한다.
- 구간별 TPS와 응답시간 목표는 예상 매장·사용자 규모, 사용자 관점의 허용 지연, 기준 성능 측정, 허용 오류율과 피크 대응 여유 용량을 근거로 확정한다.
- 현재 구현의 기준 측정 결과를 그대로 목표로 채택하지 않으며, 확정 전 수치는 목표값으로 사용하지 않는다.
- 목표를 확정한 뒤 k6 Threshold로 `p95`·`p99` 응답시간과 오류율을 자동 판정하고 같은 환경에서 반복 검증한다.
- 성공한 주문 데이터가 유실되지 않고 재시도와 중복 이벤트가 업무 효과를 중복 발생시키지 않는지 검증한다.
- Consumer Group별 Retry Topic 재시도와 DLT 격리, 기존 `eventId`를 유지한 운영자 수동 재처리가 업무 효과를 중복 발생시키지 않는지 검증한다.
- 애플리케이션 인스턴스 장애 시 다른 인스턴스가 약 30초 안에 요청 처리를 이어받는 것을 목표로 한다.
- Kafka 장애가 이미 완료된 주문을 실패로 바꾸지 않는지 검증한다.
- Redis 장애 시 PostgreSQL 집계 조회로 대체되는지 검증한다.
- PostgreSQL 장애 중 실제로 성공하지 않은 요청을 성공으로 응답하지 않는지 검증한다.
- Requirements와 README에는 검증 영역과 핵심 결과를 요약하고, 세부 테스트 케이스와 실행 증거는 테스트 코드, k6 결과와 별도 검증 자료에 둔다.

이 방향은 평균 응답시간만 측정하지 않고 지연 분포, 데이터 정확성, 중복 효과와 장애 격리를 함께 검증한다. 대신 부하 환경과 실패 주입을 준비하는 비용이 커지고, 환경 사양을 기록하지 않으면 결과를 재현하기 어렵다.

## Verification Evidence (2026-07-18)

- 상세 결과: [`../system-test-report.md`](../system-test-report.md)
- 정상 다중 인스턴스 실행에서 기능 실패와 예상하지 않은 5xx 없이 11개 정합성 불변식이 통과했다.
- 애플리케이션 한 인스턴스 종료와 Kafka 중단·복구 후에도 11개 정합성 불변식이 통과하고 미처리·실패 Outbox가 남지 않았다.
- 이번 처리량과 `p95`·`p99`는 로컬 단일 실행 기준선으로만 채택하며 Threshold나 SLO로 승격하지 않는다.
- Kafka health 복구 측정값은 로컬 하네스 관측값이며 운영 RTO를 확정하는 근거로 사용하지 않는다.
- 직접 port 순환 방식에서는 종료된 인스턴스로 향한 연결 실패가 발생하므로, 서비스 가용성 목표는 Load Balancer와 재시도 정책을 포함한 후 결정한다.

## Options Considered

- Option: 서비스 시간대별 k6 부하 검증과 계층별 정확성·장애 검증을 함께 수행한다.
- Rationale: 아침·점심 피크와 순간 주문 집중의 부하 형태가 다르고, 성능이 양호하더라도 데이터 유실이나 중복 효과가 발생할 수 있으므로 구간별 처리량과 정확성을 함께 입증해야 한다.
- Trade-offs: 테스트 데이터, 장애 주입과 결과 분석 범위가 커진다.
- Decision: Accepted.

- Option: 시간대별 부하 모델을 정의하고 기준 성능을 측정한 뒤 최종 Threshold를 확정한다.
- Rationale: 근거 없는 응답시간 숫자를 피하면서도 현재 구현 성능에 목표를 맞추는 순환 논리를 방지하고, 사용자 기대와 피크 여유 용량을 함께 반영할 수 있다.
- Trade-offs: 기준 측정과 목표 검증을 분리하여 최소 두 단계로 실행해야 하고, 환경과 데이터 조건을 고정해야 비교가 가능하다.
- Decision: Accepted.

- Option: 문서에는 검증 요약만 두고 상세 증거는 실행 가능한 테스트와 결과 자료에 둔다.
- Rationale: 요구사항과 README의 가독성을 유지하면서 검증 근거를 재현 가능한 형태로 보존한다.
- Trade-offs: 독자가 요약에서 상세 증거로 이동할 수 있도록 링크와 실행 방법을 관리해야 한다.
- Decision: Accepted.

## Proposed

- 애플리케이션 로그를 구조화된 JSON 형식으로 기록한다.
- Prometheus와 Grafana로 처리량, 지연시간, 오류, Outbox 적체와 Consumer 지연을 관찰한다.
- 단위, 통합, 동시성, Kafka 장애, Redis 장애, E2E와 부하 테스트를 계층화한다.
- 각 서비스 시간대 시나리오에 준비, 지속 부하와 회복 구간을 두고, 구체적인 실행 시간은 정량 부하 모델과 함께 확정한다.
- 동일 조건으로 기준 측정을 반복하고 중앙값과 변동 폭을 기록한 뒤 최종 Threshold 후보를 정한다.
- Docker 다중 컨테이너를 주 검증 환경으로 사용하고 AWS 일부 배포로 다중 인스턴스 동작을 보조 검증한다.
- Outbox 실패, Consumer 중복, Redis 장애와 애플리케이션 인스턴스 종료를 장애 주입 시나리오로 사용한다.

Prometheus·Grafana와 실제 AWS 검증 범위는 잠정 사항이며 Accepted로 승격하지 않는다.

## Rejected Alternatives

- Alternative: 모든 테스트 케이스와 상세 실행 결과를 Requirements 또는 README에 직접 기록한다.
- Reason: 문서 분량이 과도해지므로 요구사항 문서에는 검증 범위와 핵심 결과만 두고 상세 내용은 테스트 코드와 별도 결과 자료에 보관하기로 명시적으로 정했다.

## Downstream Impact

- Domain: 직접적인 책임 변경은 없지만 각 도메인의 불변식과 중복 효과 방지 조건을 검증 가능하게 표현해야 한다.
- Specification: 성공·실패 조건, 반영 시간과 관찰 가능한 응답을 테스트 가능하게 명세해야 한다.
- Issue: 기능 테스트, 동시성, 이벤트 장애, 캐시 대체, 부하와 관측성 작업을 구분해 추적해야 한다.
- Implementation: 지연시간과 적체를 측정할 지점, 장애 대체 경로와 재현 가능한 테스트 데이터를 제공해야 한다.
- Test: 서비스 시간대별 기준 성능과 확정된 TPS·요청 분포별 Threshold를 구분하여 기록하고, 백분위 응답시간, 오류율, 무유실, 중복 효과 방지, Retry Topic·DLT와 수동 재처리, 10초 인기 반영과 장애 격리를 검증해야 한다.
- Pull Request / Review: 성능 수치의 측정 환경, 실패 테스트의 실제 단언과 제출된 증거가 결정 기준을 충족하는지 확인해야 한다.

## Assumptions

- 추가로 채택한 미검증 가정은 없다.

## Open Decisions

- 부하 테스트의 허용 오류율과 최소 성공률.
- 아침 시간대 피크와 점심 시간대 피크의 정확한 시간 범위, 예상 사용자·매장 규모와 구간별 TPS.
- 각 부하 구간의 주문·조회 비율, `p95`·`p99` 응답시간 목표.
- 기준 성능 대비 확보할 피크 여유 용량과 Threshold 확정 규칙.
- 각 부하 단계의 실행 시간, 반복 횟수, 준비와 회복 구간.
- 테스트 데이터 분포, Hot key와 사용자·메뉴 집중 조건.
- 부하 발생기와 서버의 하드웨어 사양.
- Kafka, Redis와 PostgreSQL 장애의 정량 복구 목표.
- 모니터링 지표별 경고 임계값과 알림 경로.
- AWS에서 실제 수행할 검증 시나리오와 구성 범위.
- 인기 메뉴 10초 반영에서 정상 상황, 측정 시작점과 종료점의 정의.
- 다중 인스턴스 검증에서 사용할 Load Balancer와 연결 실패 재시도 정책.
- Outbox 선점 분포·적체 추이, PostgreSQL connection, Consumer lag, Retry Topic·DLT를 자동 수집하는 방법.
