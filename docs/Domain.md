# Domain

## 목적

확정된 요구사항과 Accepted Decision을 도메인 경계, 핵심 모델, 불변식과 의존 관계로 정리합니다. 이 문서는 업무 개념과 책임을 정의하며 API, 데이터 모델, 메시지 계약과 인프라 상세 설계는 후속 Specification 및 구현 단계에 맡깁니다.

## 현재 상태

- 요구사항의 핵심 업무 정책과 장애 경계가 확정되어 Domain 및 Specification 작업을 진행할 수 있습니다.
- 주요 아키텍처 Decision은 모두 `Accepted`입니다.
- 운영 수치와 상세 계약 일부는 근거가 없어 `Open`이며, Requirements Readiness는 `Needs Clarification`입니다.
- 기존 Domain 문서의 미정 자리표시자는 현재 Requirements 및 Accepted Decision에 맞게 갱신했습니다. 기존 문서와 확정 근거 사이에 새로운 정책 결정이 필요한 충돌은 확인되지 않았습니다.

## 도메인 범위

### 포함 범위

- 주문 가능한 커피 메뉴와 현재 가격 조회
- 인증된 사용자 식별값에 연결된 포인트 충전과 포인트 결제
- 하나 이상의 주문 항목으로 구성된 주문의 가격 확정과 완료
- 주문 완료 사실의 발행 의도 보존과 비동기 후속 처리 연결
- 완료 주문을 기반으로 한 인기 메뉴 집계와 조회
- 주문 내역의 외부 데이터 수집 플랫폼 전달

### 제외 범위

- 전체 인증 시스템의 구현
- 원재료별 수량, 레시피와 재고 관리
- 메뉴의 물리적 삭제
- 포인트 만료와 사용자 간 양도
- 주문 취소, 결제 취소와 환불
- 주문 외 포인트 충전·메뉴 변경 이벤트 발행
- 실제 데이터 수집 플랫폼의 미제공 계약
- API, ERD, 저장 구조, 메시지 채널과 캐시 키 등의 상세 설계

## 도메인 경계와 책임

### 메뉴

- 주문에 사용할 수 있는 메뉴의 식별값, 이름, 현재 가격과 `주문 가능` 또는 `주문 불가` 상태를 소유합니다.
- 메뉴 목록 조회에 필요한 정보를 제공합니다.
- 주문 시점의 가격 계산과 주문 가능 여부 판단에 권위 있는 메뉴 정보를 제공합니다.
- 현재 범위에서 판매 중지는 메뉴를 물리적으로 삭제하지 않고 `주문 불가` 상태로 변경하여 표현합니다.
- 원재료 재고나 레시피 수량은 소유하지 않습니다.

### 사용자 포인트

- 인증된 사용자 식별값에 연결된 포인트 잔액을 소유합니다.
- `1원 = 1P`로 유효한 충전금액을 반영하고, 충분한 잔액이 있을 때 주문금액을 차감합니다.
- 동시 충전과 차감에서도 잔액 유실, 이중 사용과 음수 잔액이 발생하지 않도록 합니다.
- 포인트는 만료되지 않으며 다른 사용자에게 양도할 수 없습니다.

### 주문 및 결제

- 인증된 사용자, 일회성 주문 토큰과 하나 이상의 주문 항목을 기준으로 주문 시도 하나를 식별합니다.
- 메뉴의 현재 가격과 수량으로 총 결제금액을 서버에서 계산하고 각 주문 항목에 단가 스냅샷을 보존합니다.
- 모든 메뉴가 존재하고 주문 가능하며 포인트 잔액이 충분할 때만 주문을 완료합니다.
- 주문, 주문 항목, 포인트 차감, 이벤트 발행 의도와 토큰 결과의 원자적 성공 경계를 조정합니다.
- 결제 수단은 포인트뿐이며 취소와 환불 수명주기는 갖지 않습니다.

### 주문 이벤트 발행

- 완료된 주문 하나를 나타내는 `OrderCompleted`라는 논리적 도메인 이벤트를 생성합니다.
- 주문 완료와 외부 이벤트의 실제 발행·전달 상태를 분리합니다.
- 주문 트랜잭션에는 이벤트 발행 의도까지만 포함하고 실제 발행과 후속 전달은 비동기로 수행합니다.
- 이벤트의 실제 발행 실패는 이미 완료된 주문의 결과를 바꾸지 않습니다.
- Outbox 선점, 재시도, Retry Topic과 DLT는 이 경계를 실현하고 복구하기 위한 기술적 책임이며 주문 도메인의 업무 상태가 아닙니다.

### 인기 메뉴 집계 및 조회

- 결제 완료된 `OrderCompleted` 주문만 입력으로 받아 메뉴별 총 판매 수량과 해당 메뉴가 포함된 주문 건수를 파생합니다.
- 조회 시점부터 정확히 168시간 전까지의 이동 구간을 산정하고 정렬 정책에 따라 최대 3개를 제공합니다.
- 집계 기간 내 판매 기록이 있는 메뉴는 현재 주문 불가 상태이더라도 집계에 포함합니다. 현재 주문 가능 여부는 판매 통계와 별개의 메뉴 정보입니다.
- 주문 원본을 변경하지 않으며, 주문 완료 사실에서 재구성 가능한 Projection을 소유합니다.
- 영속 집계가 기준 데이터이고 캐시는 파생된 조회 최적화 수단입니다. 캐시 장애나 오래된 값이 기준 데이터의 정확성을 대체하지 않습니다.

### 외부 데이터 수집 플랫폼 전달

- 동일한 `OrderCompleted` 이벤트에서 인증된 사용자 식별값, 복수 주문 항목과 총 결제금액을 외부 전달 형식으로 변환하여 보냅니다.
- 인기 메뉴 집계와 독립된 후속 책임이며 한쪽의 지연이나 장애가 다른 쪽을 막지 않습니다.
- 외부 플랫폼 장애, 재시도와 최종 전달 실패는 주문 완료 여부를 바꾸지 않습니다.
- 외부 플랫폼의 실제 인터페이스와 장애 처리 계약은 제공되지 않았으며 현재 범위에서는 Mock API 또는 테스트 코드로 대체할 수 있습니다.

## 핵심 모델

### Entity

- **Menu**: `MenuId`로 식별되며 이름, 현재 가격과 `주문 가능` 또는 `주문 불가` 상태를 갖습니다.
- **PointAccount**: `UserId`로 식별되는 사용자의 포인트 잔액을 소유하고 충전과 조건부 차감을 수행합니다.
- **Order**: `OrderId`로 식별되며 인증된 사용자, 주문 항목, 총 결제금액, 완료 시각과 완료 상태를 소유합니다.
- **OrderItem**: Order에 속하며 메뉴 식별값, 수량과 주문 시점의 단가 스냅샷을 가집니다. Order 밖에서 독립된 수명주기를 갖지 않습니다.
- **OrderToken**: 주문 시도 하나를 식별하고 한 요청만 처리 권한을 얻도록 하며, 확정된 성공 또는 업무 실패 결과를 재시도에 재사용합니다.

`OrderCompleted`는 변경되는 Entity가 아니라 완료된 주문의 불변 사실을 표현하는 Domain Event입니다. 인기 메뉴 집계 결과는 주문 원본에서 파생되는 Projection이며 독립적인 원본 업무 Entity로 간주하지 않습니다.

### Value Object

- **UserId, MenuId, OrderId, EventId**: 각 경계에서 대상을 식별하는 값입니다.
- **Money / PointAmount**: 가격, 충전금액, 결제금액과 포인트 수량을 표현합니다. 충전금액은 1 이상의 정수이고 `1원 = 1P`를 적용합니다.
- **MenuAvailability**: 메뉴의 `주문 가능` 또는 `주문 불가` 상태를 표현합니다.
- **Quantity**: 주문 항목의 수량을 표현하는 1 이상의 정수입니다. 현재 업무상 최대 주문 수량은 없지만 요청 자료형 또는 안전한 금액 계산 범위를 초과할 수 없습니다.
- **UnitPriceSnapshot**: 주문 시점에 서버가 확인한 메뉴 단가로, 주문 완료 후 현재 메뉴 가격과 분리되어 보존됩니다.
- **OrderLineRequest**: 메뉴 식별값과 수량으로 구성된 주문 입력 항목입니다.
- **PopularMenuMetric**: 메뉴별 총 판매 수량과 해당 메뉴가 포함된 주문 건수를 함께 표현합니다.
- **TimeWindow**: 조회 시점과 그 시점부터 정확히 168시간 전까지의 절대 시각 구간을 표현합니다.

### Aggregate와 Aggregate Root

- **Menu Aggregate — Root: Menu**: 메뉴 정보와 주문 가능 여부의 일관성을 책임집니다.
- **PointAccount Aggregate — Root: PointAccount**: 한 사용자의 잔액과 충전·차감 불변식을 책임집니다.
- **Order Aggregate — Root: Order**: OrderItem의 구성, 서버 계산 총액, 단가 스냅샷과 완료 사실의 일관성을 책임집니다.
- **OrderToken Aggregate — Root: OrderToken**: 주문 시도의 처리 권한, 주문 내용의 동일성, 유효기간과 확정 결과 재사용을 책임집니다.

인기 메뉴 Projection은 읽기 목적의 파생 모델입니다. 집계 저장을 위한 시간 버킷이 미확정이므로 별도의 Aggregate 구조를 확정하지 않습니다. 주문 이벤트의 발행·소비 상태도 Order Aggregate의 업무 상태에 포함하지 않습니다.

### Domain Service가 필요한 책임

- **주문 가격 산정 정책**: 여러 메뉴의 현재 가격과 주문 수량을 결합해 총액과 단가 스냅샷을 산정하는 순수 업무 규칙입니다. Menu나 Order 하나에만 책임을 귀속하기 어려운 경우 Domain Service로 둡니다.
- **인기 메뉴 순위 정책**: 168시간 구간의 Projection에서 총 판매 수량, 주문 건수와 메뉴 ID의 우선순위를 차례로 적용하여 최대 3개를 선택하는 순수 업무 규칙입니다.

주문 토큰의 처리 권한 획득, 메뉴 조회, 포인트 조건부 차감, Aggregate 저장과 발행 의도의 원자적 확정은 여러 경계를 조정하는 애플리케이션 책임입니다. 외부 주문 내역 형식 변환과 실제 전달도 통합 경계의 책임이며 Domain Service로 취급하지 않습니다.

### Repository의 도메인 관점 역할

- **Menu Repository**: 식별값으로 권위 있는 메뉴를 찾고 조회 가능한 메뉴 목록을 제공합니다.
- **PointAccount Repository**: 사용자 포인트 계정을 찾고 동시 요청에서도 원자적 충전과 잔액 조건부 차감 결과를 제공합니다.
- **Order Repository**: 완료된 Order와 포함된 OrderItem을 하나의 Aggregate로 보존하고 조회합니다.
- **OrderToken Repository**: 토큰의 유효성, 처리 권한, 최초 주문 내용과 확정 결과를 일관되게 보존하고 동일 토큰·동일 주문 내용의 재시도에 기존 결과를 제공합니다.
- **Order Event Publication Gateway**: Order와 함께 발행 의도가 유실되지 않도록 보존하고, 실제 발행 수명주기는 Order와 분리하여 추적합니다.
- **Popular Menu Projection Repository**: 완료 이벤트의 업무 효과를 중복 반영하지 않고 영속 집계를 갱신하며, 특정 조회 시점의 168시간 집계 결과를 제공합니다.

Repository는 도메인 관점의 저장·조회 역할만 나타냅니다. 테이블, SQL, 인덱스, 브로커 Topic, Consumer Group 식별자와 캐시 Key는 이 문서에서 정하지 않습니다.

## 불변식과 상태

### 메뉴

- 메뉴는 `주문 가능` 또는 `주문 불가` 상태를 가지며 현재 범위에서 물리적으로 삭제하지 않습니다.
- 판매 중지는 `주문 불가` 상태로 표현하여 완료 주문과 인기 메뉴가 참조하는 `MenuId`를 유지합니다.
- 메뉴 상태 변경은 이미 완료된 주문과 주문 시점에 보존된 메뉴 정보 스냅샷을 변경하지 않습니다.

### 포인트

- 충전금액은 자료형 범위 안의 1 이상의 정수여야 하며 업무상 최대 한도는 없습니다.
- 존재하지 않거나 유효하지 않은 사용자의 충전과 주문 요청은 거부합니다.
- 사용자 식별값은 인증된 사용자 정보에서 얻으며 요청 Body의 값을 신뢰하지 않습니다.
- 포인트는 만료되지 않고 사용자 간 양도를 지원하지 않습니다.
- 차감은 잔액이 주문금액 이상일 때만 성공하며 결과 잔액은 음수가 될 수 없습니다.
- 충전과 차감이 동시에 실행되면 먼저 반영된 연산을 기준으로 후속 연산의 성공 여부가 결정됩니다. 충전 포인트를 확실히 사용할 주문은 충전 완료 후 시도해야 합니다.

### 주문과 결제

- 주문은 하나 이상의 주문 항목을 가져야 하며 각 항목은 메뉴 ID와 수량을 가집니다. 한 주문에 포함할 수 있는 주문 항목의 업무상 최대 개수는 두지 않습니다.
- 각 주문 항목의 수량은 1 이상의 정수여야 합니다. 현재 업무상 최대 주문 수량은 두지 않으며, 요청 자료형 또는 안전한 금액 계산 범위를 초과하는 값은 거부합니다.
- 한 주문 요청에 동일한 메뉴 ID가 둘 이상의 주문 항목으로 중복되면 합산하지 않고 요청 전체를 거부합니다.
- 지나치게 큰 요청의 제한은 주문 Aggregate의 업무 규칙이 아니라 API 요청 크기 및 운영 보호 기준으로 별도 관리합니다.
- 주문 가능 상태의 메뉴만 신규 주문에 포함할 수 있습니다. 주문 항목 중 하나라도 주문 불가 상태이면 주문 전체를 거부합니다.
- 서버가 주문 시점의 현재 메뉴 가격과 수량으로 총액을 계산하며 클라이언트가 제시한 가격을 권위 있는 값으로 사용하지 않습니다.
- 주문 완료 시 각 항목의 단가 스냅샷과 계산된 총 결제금액은 서로 일치해야 합니다.
- 존재하지 않거나 주문할 수 없는 메뉴가 하나라도 포함되거나 잔액이 부족하면 전체 주문이 실패합니다.
- 실패한 주문 시도에는 Order, OrderItem, 포인트 차감과 이벤트 발행 의도가 부분적으로 남지 않습니다.
- 처리 권한을 얻은 요청의 포인트 차감, Order와 OrderItem 생성, 이벤트 발행 의도 기록과 토큰 결과 확정은 하나의 로컬 트랜잭션에서 함께 성공하거나 실패합니다.
- 완료된 Order는 현재 범위에서 취소·환불 상태로 전이하지 않습니다. 업무 실패는 Order를 생성하지 않고 OrderToken의 확정 실패 결과로 남습니다.

### 일회성 주문 토큰

- 서버가 발급한 토큰 하나는 주문 시도 하나에만 대응합니다.
- 동일 토큰의 동시 요청 중 하나만 처리 권한을 얻고, 나머지는 확정된 동일 결과를 사용하여 주문과 포인트 차감의 중복 효과를 만들지 않습니다.
- 동일 토큰과 동일 주문 내용의 재시도에만 확정된 기존 결과를 반환합니다.
- 이미 사용된 토큰에 이전과 다른 주문 내용이 들어오면 충돌로 거부하며 새 Order나 포인트 차감을 만들지 않습니다.
- 사용하지 않은 토큰의 유효기간은 10분이고 확정 결과의 보관기간은 24시간입니다.
- 토큰은 개념적으로 `사용 가능 → 처리 중 → 성공 확정 또는 업무 실패 확정`으로 진행하며 확정 결과는 되돌리지 않습니다.
- 유효기간이 지난 토큰은 `만료` 업무 실패로, 발급되었거나 보관된 사실이 없는 토큰은 `미존재` 업무 실패로 구분합니다. 두 경우 모두 주문 처리 권한을 부여하지 않고 새 Order나 포인트 차감을 만들지 않습니다.
- `만료`와 `미존재`의 HTTP 상태와 오류 응답 형식은 Specification에서 정합니다.

### 주문 완료와 이벤트 발행

- Order가 결제 완료되어야만 `OrderCompleted` 사실과 발행 의도를 확정할 수 있습니다.
- 주문 완료는 외부 브로커 발행이나 외부 플랫폼 전달의 성공을 기다리지 않습니다.
- 이벤트 발행 의도는 Order와 원자적으로 보존되지만 실제 발행 상태는 Order 상태와 별도로 진행됩니다.
- 전달은 중복될 수 있으므로 각 후속 업무는 같은 `eventId`의 업무 효과를 한 번만 반영해야 합니다.
- 발행 재시도, 최종 발행 실패, Consumer별 재시도와 DLT 격리는 기술적 전달 상태이며 완료된 Order를 실패로 되돌리지 않습니다.

### 인기 메뉴

- 결제까지 성공한 `OrderCompleted` 주문만 집계하며 완료되지 않은 주문은 제외합니다.
- 조회 시점 기준 최근 168시간 내 완료 주문에 포함된 메뉴는 현재 주문 불가 상태이더라도 인기 메뉴 집계에서 제외하지 않습니다.
- 완료된 한 주문에서 각 메뉴는 하나의 주문 항목으로만 표현되며, 해당 메뉴의 주문 건수에 한 번 기여하고 주문 수량 전체가 판매 수량에 반영됩니다.
- 업무 기준 시간대는 `Asia/Seoul`이고 저장·이벤트 시각은 UTC로 관리합니다.
- 조회 대상은 달력 날짜가 아니라 조회 시점부터 정확히 168시간 전까지의 이동 구간입니다.
- 총 판매 수량 내림차순, 해당 메뉴가 포함된 주문 건수 내림차순, 메뉴 ID 오름차순으로 정렬합니다.
- 최대 3개를 반환하며 집계 결과가 3개 미만이면 실제 결과만, 결과가 없으면 빈 목록을 반환합니다.
- 정상 상황에서는 주문 완료 후 10초 이내에 최신 판매 수량이 조회 결과에 반영되어야 합니다.
- 영속 Projection 갱신은 같은 이벤트의 재전달로 중복 반영되지 않아야 합니다. 영속 집계 갱신 뒤 캐시 무효화가 실패하면 집계를 다시 더하지 않고 무효화만 재시도합니다.

## 트랜잭션과 비동기 경계

- **포인트 충전:** PointAccount의 원자적 증가 하나가 트랜잭션 경계입니다.
- **주문 완료:** OrderToken 처리 권한과 결과, PointAccount 차감, Order와 OrderItem, 이벤트 발행 의도가 하나의 로컬 원자적 경계입니다.
- **이벤트 실제 발행:** 주문 완료 트랜잭션 밖에서 비동기로 수행하며 실패해도 주문 성공을 변경하지 않습니다.
- **후속 업무:** 외부 플랫폼 전달과 인기 메뉴 집계는 서로 독립된 소비·재시도·장애 경계를 갖습니다.
- **인기 메뉴 반영:** 영속 Projection 갱신과 중복 처리 방지는 하나의 원자적 경계입니다. 캐시 무효화는 별도 기술 자원이므로 같은 트랜잭션에 포함하지 않고, 성공할 때까지 후속 처리 완료를 확정하지 않습니다.

## 관계와 의존성

- **인증된 사용자 정보 → 포인트·주문:** 포인트와 주문은 외부 인증 경계가 제공한 UserId에 의존하지만 인증 시스템 자체를 소유하지 않습니다.
- **메뉴 → 주문:** 주문은 완료 시점에 메뉴의 현재 가격과 주문 가능 여부를 참조합니다. 완료 후에는 OrderItem에 보존된 주문 시점의 메뉴 정보와 단가 스냅샷이 주문 원본이며 이후 메뉴 가격이나 주문 가능 상태 변경에 영향받지 않습니다.
- **메뉴 → 인기 메뉴 Projection:** 인기 메뉴의 판매 통계는 완료 주문에 의존하며 메뉴의 현재 주문 가능 상태로 과거 판매 기록을 제거하지 않습니다. 현재 주문 가능 여부는 필요할 경우 조회 응답에서 별도의 메뉴 정보로 결합합니다.
- **포인트 ↔ 주문:** 서로 다른 Aggregate이지만 주문 완료 업무에서는 조건부 차감과 Order 생성이 하나의 로컬 트랜잭션으로 묶입니다. PointAccount는 주문 상세를 소유하지 않고 Order는 포인트 잔액을 소유하지 않습니다.
- **주문 → 주문 완료 이벤트:** 이벤트는 완료된 Order에서 생성되는 불변 사실입니다. 발행 상태는 Order의 완료 상태와 분리됩니다.
- **주문 완료 이벤트 → 인기 메뉴 Projection:** Projection은 완료 이벤트에 의존하는 파생 읽기 모델이며 Order를 변경하지 않습니다. 정확성 검증과 장애 복구의 최종 근거는 영속 주문·집계 데이터입니다.
- **주문 완료 이벤트 → 외부 플랫폼 전달:** 전달 경계는 주문 완료 사실에 의존하지만 외부 플랫폼의 가용성은 주문 성공의 선행 조건이 아닙니다.
- **기준 데이터 → 캐시:** 주문, 포인트와 인기 메뉴 영속 집계가 기준 데이터이고 캐시는 그 결과에서 파생됩니다. 캐시 장애 시 기준 집계 조회로 대체합니다.

의존 방향은 `메뉴·인증 정보·포인트 → 주문 완료 → OrderCompleted → 인기 메뉴 Projection / 외부 플랫폼 전달`입니다. 인기 메뉴와 외부 전달은 주문이나 포인트를 역으로 변경하지 않으며 서로에게도 의존하지 않습니다.

## 외부 시스템과 기술 경계

- **인증 정보 제공자:** UserId를 신뢰할 수 있는 인증 정보로 제공합니다. 제공 주체, 토큰 형식, 필수 Claim과 검증 계약은 Open입니다.
- **데이터 수집 플랫폼:** 완료 주문 내역을 받는 외부 시스템입니다. 인터페이스, 가용성과 장애 처리 계약은 Open이며 장애는 주문 성공과 격리됩니다.
- **영속 저장 기술:** 원자적 주문·포인트·발행 의도와 기준 집계를 보존하는 수단입니다. PostgreSQL은 Accepted된 구현 선택이지만 도메인 Entity가 아닙니다.
- **이벤트 전달 기술:** 완료 이벤트를 독립된 후속 업무에 전달하는 수단입니다. Kafka, Outbox Publisher, Retry Topic과 DLT는 Accepted된 구현·운영 책임이지만 업무 Aggregate가 아닙니다.
- **조회 최적화 기술:** 인기 메뉴 Projection 조회를 가속하고 정기 Refresh를 조정합니다. Redis, 캐시 무효화와 분산 락은 기술적 책임이며 기준 데이터나 도메인 불변식의 소유자가 아닙니다.
- **API 요청 보호:** 지나치게 큰 요청을 제한하는 요청 크기와 운영 보호 기준은 외부 입력 경계의 기술적 책임이며 주문 항목 개수의 업무상 상한으로 취급하지 않습니다.

## 확정 정책과 Open 사항

### 확정 정책

- 포인트 충전·차감, 서버 가격 계산, 단가 스냅샷, 전체 주문 실패 조건과 주문 완료 원자 경계
- 하나 이상의 주문 항목, 주문 항목 개수의 업무상 최대 없음과 동일 메뉴 ID 중복 요청의 전체 거부
- 주문 항목 수량의 1 이상 정수 조건, 업무상 최대 수량 없음과 안전한 금액 계산 범위 초과 거부
- 메뉴의 `주문 가능`·`주문 불가` 상태, 주문 불가 메뉴가 하나라도 포함된 전체 주문의 거부와 완료 주문 스냅샷 불변성
- 메뉴 물리적 삭제 미지원과 판매 중지의 `주문 불가` 상태 처리
- 일회성 주문 토큰의 처리 권한, 10분 유효기간, 성공·업무 실패 결과의 재사용과 24시간 보관
- 동일 토큰·동일 주문 내용의 기존 결과 반환과 동일 토큰·다른 주문 내용의 충돌 거부
- 만료 토큰과 미존재 토큰의 서로 다른 업무 실패 구분
- 주문 완료와 이벤트 실제 발행·전달 상태의 분리
- `at-least-once` 전달에서 `eventId`를 유지한 후속 업무별 중복 효과 방지와 장애 격리
- `OrderCompleted`, 최근 168시간, 시간대, 정렬과 최대 반환 수를 적용한 인기 메뉴 정책
- 현재 주문 불가 메뉴도 최근 168시간 내 완료 주문의 판매 기록이 있으면 인기 메뉴 집계에 포함하는 정책
- 영속 집계를 기준 데이터로, 캐시를 파생 데이터로 사용하는 경계와 캐시 장애 시 대체 조회
- 정상 상황의 주문 완료 후 이벤트 발행 3초 이내 및 인기 메뉴 조회 반영 10초 이내 목표

### Open / Downstream Concerns

- 데이터 수집 플랫폼 전달 완료까지의 end-to-end 최대 지연시간
- Consumer 재시도 횟수와 간격
- DLT 보관기간과 명시적 폐기 승인 절차
- Redis 정기 Refresh 주기와 TTL
- 인기 메뉴 집계 저장을 위한 시간 버킷 단위
- 이벤트 실제 타입명, Topic명, 직렬화 형식과 스키마 버전 관리 방식
- 인증 정보 제공 주체, 토큰 형식과 필수 Claim
- 서비스 시간대별 부하, 요청 분포, 허용 오류율과 `p95`·`p99` 성능 Threshold
- API 요청 크기와 운영 보호 기준
- 메뉴 상태 변경 주체와 변경 계약
- 인기 메뉴 응답에 현재 주문 가능 여부를 포함할지와 표현 방식
- 데이터 수집 플랫폼의 실제 인터페이스와 장애 처리 계약
- 주문 토큰 정리 주기와 장기 감사 데이터 보관 정책

## Specification 전달 사항

- 인증된 사용자 식별값을 받는 경계와 사용자 유효성 실패 계약
- 포인트 충전, 주문 토큰 발급·사용·재시도, 동일 토큰의 주문 내용 일치 검증, 충돌 거부, 만료·미존재 구분과 주문 성공·업무 실패의 입력·응답·오류 계약
- 복수 주문 항목, 동일 메뉴 ID 중복 거부, 1 이상의 정수 수량, 안전한 금액 계산 범위 검증, 서버 가격 계산, 단가 스냅샷과 총액의 표현
- 메뉴의 `주문 가능`·`주문 불가` 상태, 주문 불가 메뉴의 전체 주문 실패와 상태 변경 후 완료 주문 스냅샷 불변성의 표현
- 메뉴 물리적 삭제 미지원과 판매 중지 상태 변경 계약
- `OrderCompleted`의 실제 계약: 이벤트·주문 식별값, 사용자 식별값, 복수 항목, 총 결제금액, 발생 시각과 스키마 호환성
- 인기 메뉴 응답, 정렬, 최대 3개, 빈 결과, 168시간 경계와 집계 기준 시각, 현재 주문 가능 여부의 포함 여부와 표현
- 외부 데이터 수집 플랫폼 전달 계약과 Mock 대체 경계
- 주문 항목 개수의 업무상 상한과 구분되는 API 요청 크기 및 운영 보호 기준
- 확정되지 않은 수치와 계약은 Open으로 유지하고 근거가 확보된 뒤 별도 결정으로 확정
